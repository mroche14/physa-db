# ADR-0001: Project tracking — GitHub Issues + GitHub Projects v2, with a static JSON snapshot for the dashboard

- **Status:** Proposed
- **Date:** 2026-04-20
- **Context issue:** _(to be opened once repo is pushed)_

## Context

The founder asked whether to use a local SQL DB, a NoSQL store, GitHub Issues, or "something else" to track the project state (features, todos, progress) so that:

1. Contributors and especially the founder (Marvin) can see what is going on at a glance.
2. AI agents can read and write tasks programmatically.
3. The tracker is **self-contained in the repo** — no external SaaS account, no credit card.
4. A small app on **GitHub Pages** surfaces the state publicly.

Candidates considered:

| Option | Pros | Cons |
|--------|------|------|
| **GitHub Issues + Projects v2** | Native, free, no infra, `gh` CLI works everywhere, structured fields in Projects v2, rich history, notification system | Limited custom schema; tied to GitHub as platform (but so is the repo); rate-limited API |
| **SQLite committed to repo** | ACID, local, queryable via SQL | Binary diffs are noisy in git, contention on concurrent writes, dashboard needs a CGI or WASM SQLite reader |
| **JSON/YAML files per task in repo** | Git-diffable, human-readable, versioned, review via PR | Assignment, comments, status changes all need a PR — heavyweight for a simple status flip |
| **External SaaS (Linear, Jira, Notion)** | Rich feature set | Violates "self-contained in repo" constraint; account barrier for contributors |
| **Custom Rust tracker + sled DB** | Dogfood our own storage | Circular dependency: can't track building the DB with the DB we're building |

## Decision

We use **GitHub Issues as the primary system of record**, organised through **GitHub Projects v2** (for status/priority/sprint fields), and **render a static dashboard** on GitHub Pages backed by a `dashboard/data/state.json` file that a CI workflow regenerates on every issue event.

- **Issues** = tasks, bugs, features, research items. One issue per unit of work.
- **Projects v2** = the board view + structured fields (`Status`, `Priority`, `Area`, `Sprint`, `Estimate`).
- **Labels** = categorical metadata (see `AGENTS.md` §6 for the canonical set).
- **Milestones** = roadmap markers (`M0` … `M6` from `ROADMAP.md`).
- **Markdown files under `docs/`** = durable knowledge (research notes, ADRs, benchmark writeups). NOT a duplicate tracker.
- **`dashboard/`** = a tiny SPA (vanilla JS + a CSS framework, TBD) that reads `dashboard/data/state.json`. The JSON is generated by a GitHub Action on `issues`, `pull_request`, and `project_v2_item` events, and committed to the `gh-pages` branch.

### Why the JSON snapshot?

Two reasons:

1. **No GitHub API rate limits for dashboard visitors.** The SPA reads a static file, so we can serve millions of views without hitting any quota.
2. **Offline / archival.** `state.json` is versioned in `gh-pages` history, so we can replay project state at any past moment.

## Consequences

**Positive:**
- Zero external infra: no DB to backup, no SaaS to pay, no secrets beyond `GITHUB_TOKEN`.
- AI agents already know the `gh` CLI; no custom API to learn.
- Contributors can use the GitHub UI they already know.
- Dashboard is fast (static file) and free to host.

**Negative:**
- GitHub outages = tracker outages. Acceptable (we're on GitHub anyway).
- Projects v2 GraphQL API is rate-limited; the CI snapshot must paginate correctly.
- Structured queries ("show me all p0 bugs in area:cluster opened in the last 30 days") require the snapshot to pre-compute indexes OR the dashboard to filter client-side over the full JSON. Acceptable while the issue count is < 10k.

## Alternatives considered

- **SQLite in repo** — rejected: binary diffs are hostile to code review, and no good reason to pick this over a flat JSON.
- **Self-hosted tracker (Gitea issues, OpenProject, YouTrack)** — rejected: violates self-contained-in-repo.
- **Task files as JSON/YAML in repo, one per task** — rejected: status transitions would require a PR per flip, which is too heavy.
- **Write our own tracker in Rust backed by sled** — rejected: bootstrapping problem (we need a tracker before we can build the DB that backs the tracker) and feature-creep risk.
